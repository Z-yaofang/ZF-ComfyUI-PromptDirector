# 完整口播稿

括号中的内容是操作提示，不需要读出来。整段约 18–22 分钟，可按实际结果删减。

## 00:00 先看结果

大家好，这期不讲一套新模型，也不从头搭建一套复杂工作流。我只讲一个核心：ZF 创意导演如何把零散输入组织成真正可执行的生图提示词。

我会用同一份最终提示词，分别交给本地成图模型和闭源/API模型。这样大家能清楚看到：提示词功能负责什么，模型能力又负责什么。

（展示两组画面：左边本地模型，右边闭源/API模型；不要先解释哪一张更好。）

## 01:00 五类输入

先把导演台的输入分成五类。

第一是用户提示，负责硬要求：主体、数量、身份、地点、动作、关系、关键物和文字。

第二是世界观。世界观不是必须照抄的故事正文，而是一套可以取材的角色、地点、器物、材料、事件和规则。没有世界观时，使用 Helper 提供的明确空文本路线。

第三是用途。用途回答“这张图要完成什么专业任务”，例如人物摄影、产品图、电影海报或故事分镜。

第四是视觉方法。它回答“这些内容要如何被观看”，例如决定性瞬间、人物关系场、微缩模型或双重曝光。

第五是参考图临时创意。参考图经过分析后，会临时提供一套创意机制、元素素材、构图关系和材质光线。它不是把原图像素搬过来。

可以这样记：用户提示决定必须画什么；世界观提供取材；用途决定作品任务；视觉方法组织画面；参考图提供本次创作的灵感材料。

## 03:00 两张接法图

（显示本地接法图。）

第一张是本地路线。参考图先交给本地多模态模型分析，分析结果进入参考图分析器和临时创意适配器；导演台再把用户提示、世界观、用途和参考图创意组织成任务；最后由本地文本模型把任务写成成品提示词。

（显示 API 接法图。）

第二张是 API 路线。API①只负责读图，返回结构化分析；导演台接收这份分析；API②只负责把导演台任务写成最终提示词。API②不接图片。两个 API 可以用同一服务，但不能把两个阶段混成一个节点，因为后一个阶段依赖前一个阶段的分析结果。

今天只讲这两个板块：导演台的提示词能力，以及本地和闭源/API写作、成图执行的差别。密钥、模型文件和账户信息不进入录屏。

素材库来源说明也要说清楚：插件中使用、整理和扩展的用途、创意方向、案例启发和相关提示词素材，来源于远古大呲花的素材整理与分享，B站主页在视频简介和仓库首页都有链接。插件代码、结构化整理和节点实现由本仓库完成。

## 05:00 提示词核心演示

（放大用户提示、世界观动态单选、用途与视觉方法。）

我先输入一条短提示，例如：一位成年产品设计师在清晨工作室中整理新作品集，桌面有草图、色卡和比例模型。

先选择空文本，让观众看到最少干扰时导演台怎么工作。然后选择一个现实世界观，再换成一个科幻世界观。用户明确写出的成年设计师和工作室任务应该保持稳定，世界观主要扩展地点、材料、器物和事件，不应偷偷替换用户的核心要求。

接着选择一个用途和一个视觉方法。第一组通常是主导组合，第二组可以作为补充，但不是越多越好。两个互相冲突的组合同时开启时，提示词可能出现校园、微缩、海报等不同方向的混合，这属于输入目标冲突，不一定是节点错误。

（运行一次，先展示任务列表。）

这里先看任务列表，而不是先看图片。插件代码负责建立准确数量的任务，每个任务对应一张图；语言模型负责把单个任务写成自然、完整的提示词。任务列表已经错了，就先回到用途、世界观和用户提示检查。

（展示一条最终提示词。）

最终提示词需要具体、确定、可执行。尽量不要写“可能是中文或日文”“似乎有文字”这类不确定描述，否则成图模型可能把不确定性画成乱字。需要文字时，明确写出内容；不需要文字时，写无可读文字或仅作装饰纹理。

## 09:00 参考图是创意素材库

（放大参考图分析器和临时创意适配器。）

参考图创意迁移不是洗图。参考图的 latent 不进入采样器，采样器仍然使用新的空 latent。分析器提取的是主体元素、核心机制、构图镜头、材质、色彩、光线和动作关系。

例如原图表现人物和纸上线稿重叠，重点不是必须保留某一朵荷花，而是二维线稿和三维人物重叠、创作者动作进入画面、真实物件和纸面图形并置的创意机制。新图可以保留荷花，也可以让世界观把它扩展成菊花、数字屏幕或其他媒介。关键是机制被迁移，而不是像素被复制。

如果只想测试参考图本身，可以使用空文本和单一用途；如果还接入世界观，世界观就是第二个素材库，负责扩张和替换。不要在测试时同时接多个互相冲突的用途创意，也不要把参考图 latent 接到 KSampler。

## 12:00 看最终提示词

现在把同一组输入跑完，依次查看导演台的系统提示词、任务列表和最终提示词。检查四件事：任务数量是否准确，用户硬要求是否保留，世界观扩展是否服务于用途，参考图创意是否被写成确定性的画面机制。

如果最终提示词正确但图片少格、文字乱码、微缩比例不稳定，先不要改导演台。故事分镜图本来就是一张复合图，模型是否能执行均分格、格数和一致性，属于成图模型的能力边界。视频参考帧仍建议逐张生成完整尺寸画面。

## 15:00 本地与闭源/API对比

下面只替换成图执行端，提示词、尺寸、数量和种子尽量保持一致。

本地模型的优点是离线、成本低、可控；缺点是不同模型的视觉理解、长提示词跟随和创意扩张能力差异很大。一个本地模型没有画出完整构图，不代表导演台没有生成正确提示词。

闭源/API模型通常有更强的语义理解、长文本跟随和复杂关系执行能力，尤其适合参考图创意扩张、产品图结构和复杂场景。但它有调用成本、网络依赖和服务限制。

（并排展示结果。）

比较时不要只说“哪张更漂亮”，要看三点：第一，主体和关系有没有执行；第二，参考图的创意机制有没有迁移；第三，文字、格数、材质和光线这些细节有没有按提示词落地。这样观众能理解提示词功能和模型能力是两层问题。

## 19:00 易错点与总结

第一，正常成图不要选参考图创意提取测试模式；它是隔离干扰项的诊断模式，会主动移除静态用途和视觉方法。

第二，语义创意迁移不要把原图 latent 接到采样器，否则流程就变成图生图或结构控制。

第三，空文本不是报错，而是明确的零内容路线；用户只想使用参考图和用途创意时可以直接使用。

第四，多个用途创意同时开启可能互相冲突，先保留一组确认主导方向，再逐个增加。

第五，任务列表正确、最终提示词正确，而成图模型没有执行好时，优先更换执行模型或降低任务复杂度，不要把模型问题误判成插件接线问题。

这套插件最重要的不是把提示词写得更长，而是让每个输入各司其职：用户提示负责硬要求，世界观提供素材，用途确定专业目标，视觉方法组织画面，参考图提供临时创意，写作模型把任务翻译成成品提示词，成图模型再负责执行。
